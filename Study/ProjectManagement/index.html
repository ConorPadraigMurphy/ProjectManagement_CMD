Project Management

Generic Project Management

What is a project?

→ A temporary endeavour to create a unique product

What is not a project?

→ A routine task

→ E.g. running payroll, anything operational

Characteristics of a project

→ Has a beginning and an end

→ Has goals

→ Constrained by time, cost, scope and quality

→ Preformed and completed by a group

What are projects used for?

→ To drive change and innovation

→ Enables business value creation

→ To get things done

How is a project accomplished?

TEAM

→ Must have clear goals

→ The goals must fit the mission

→ Everyone must have defined roles and responsibilities

→ Work based on deadlines or priorities

→ Communication must be clear

→ Team members must trust and respect each other

Project Goals

→ A tangible statement of what needs to be achieved

S.M.A.R.T

→ Specific, Measurable, Achievable, Relevant and Time-Bound

Example

Cost Reduction

→ A restaurant redesigns it kitchen with new ovens that allow the kitchen to make its own bread. This will reduce ingredient costs by 3% and improve food quality

Customer Experience

→ A new website that will improve customer satisfaction by 3%

Stakeholders

→ Any group or individual who can affect the achievement of the projects objective

IT Project Management

Traditional Software Development Project life cycle( SDLC )

→ The generic project life cycle tailored for software development

Waterfall SDLC

→ A series of phases that a software project passes through from beginning to end.

Advantages

→ Simplicity

→ Linear Sequential phases

→ Each phase flows on from the deliverables of the previous one

→ Similarity to traditional project management

Disadvantages

→ End users involved intermittently

→ Too much dependency on process

→ Too much emphasis on upfront documentation

→ Scope was fixed and difficult to change

→ Progress slow, releases were taking six months or more

→ Developers give the code to QA without them knowing anything about the code

→ No developer accountability

RAD Project Life Cycle

→ Rapid application development

→ A software development methodology that uses minimal planning in favour of rapid prototyping

→ Small teams compromising of developers, domain experts, customer representatives and other IT sources

→ Prototyping approach

Agile Project Life Cycles

→ Umbrella term of a set of methods and practices

→ An alternate to traditional project management

→ Based on the values and principles in the agile manifesto

→ The customer is part of the team

→ The ability to create and respond to change in order to succeed in an uncertain and turbulent environment.

→ Rapid small cycles

→ Frequent incremental releases with each release building on previous functionality

→ Thorough testing is done to ensure that software quality is maintained

The Birth Of Agile

→ In February 2001, 17 software developers met at the Snowbird, Utah resort to discuss lightweight development methods.

→ They published the manifesto for agile software development to define the approach now known as agile software development.

→ The Agile Alliance, a non-profit organisation that promotes software development according to the manifestos principles.

Famous Participants

→ The authors of the Agile Manifesto.

Summary of the Principles of Agile Methods

  1. Customer Involvement

    → The customer prioritizes what gets developed

    → The customer decides what goes into each release(not the development team)

    → The customer must work closely with development team(not just the start)

  2. Incremental Delivery

    → A system should be developed/delivered in increments and not all at once

  3. People Not Process

    → The development team should be left to deliver an increment without interference

    → It is people who write software(not processes)

    → Face to Face is better

  4. Embrace Change

    → Expect requirements to change and adapt accordingly(not fixed at the start of the project)

  5. Quality

    → Quality is the responsibility of the development team.(not the QC team of the end users)

    → QC person should be part of the development team.

  6. Maintain Simplicity

    → Simplify the design and development, eliminate complexity in the system

  7. Measuring Progress

    → Working software is the measure of progress not documentation.

Scope Requirements

Project Scope VS Product Scope

Project Scope

→ All the work that needs to be done to make a product(not just the features of the product)

→ Figuring out what works needs to be done for your project.

→ Making sure your end product does what it is supposed to do.

→ Doing the right stuff.

→ Planning, Quality, Costing, HR, etc….

Product Scope

→ The features and functions of the product or service that you and your team are building

Traditional Scope

→ Plan Driven

→ Planning is upfront at start of project

→ Requirements fixed at start of project

Agile Scope

→ Value Driven.

→ Plan/replan as you do.

→ Requirements are fuzzy at the start of the project are prioritized based in value.

Scrum

Scrum Characteristics

  1. Team Focused

    → Small cross functional teams develop software

  2. Lightweight, Inspect and Adapt Framework

    → Used when not all requirements are known up front

  3. Product Backlog

    → All future work is carried in the backlog

  4. Sprint

    → Delivers tested, production ready, incremental value

    → Sprint backlog

    → Typically 2 week work time-boxed intervals

    → Ceremonies: Daily/Weekly stand-up, Planning meetings, Review meeting and retrospective meetings

  5. The Product Owner

    → Represents the customer but is part of the scrum team

  6. The Scrum Master

    → Mentors the team on the scrum process and roles, ceremonies, artifacts and good engineering practices

  7. QC Person

    → Automates the tests with the help of the development team

  8. The Development Team

    → Develops software, manages itself, creates “Done” increments

  9. Quality

    → Everyone is responsible for quality

  10. Requirements, Architecture , Design and Development

    → Are elaborated just in time.

    The Sprint

    The Sprint Time-Box

    → The basic work process in Scrum

    → A time-boxed mini project

    → Typically a 2 week time box with a specific goal or deliverable (sprint goal)

    → No change to the goal during the sprint

    → Scope can be adjusted based on new learning

    Parts of a Sprint

    → Sprint planning meeting is held to decide what goes into sprint

    → Daily scrum meeting to report progress

    → Scrum board used to manage work

    → Burndown chart to visualise progress

    → Backlog refinement meeting(s)

    → Ends with:

    → Demo/Review

    → Retrospective

    → Optional releasable increment

Traditional Software Development Estimating

→ Time, cost and scope are estimated at the start of a project

→ Typically top down

→ Based on time

→ Delivery date calculated based on time estimates for the scope

Absolute Estimating

→ How long would it take to blow up these balloons?

The Problem with Estimating

→ Estimates are guesses

→ Typically not estimated by the person who will eventually do the work

→ Get turned into hard estimates

→ Accurate up-front estimates aren’t possible

→ Breaks, holidays, sick days are often included in estimates.

→ Time is calculated differently based on experience

→ Absolute sizing is different

Agile Estimating

→ Uses relative estimating using story points

→ Agile estimation techniques use a top down process

→ Reminds us that the estimates are just guesses

→ The person who will do the work is involved in putting the estimate together

Agile Sizing

→ Story points

→ Stories are sized relatively to each other in story points

→ The team sizes the stories in story types.

→ Team Velocity

→ The team decides how many story points a team they can do in a sprint

Extreme Programming

Waterfall vs XP

XP Core Concept

→ Software development is a fluid process

→ Requirements cannot be fully predicted from the beginning but will always change

→ Software development needs a methodology that is capable of adapting and changing requirements

XP Origins

→ Was one of the first agile methods

→ Primary catalyst that got attention to agile methods

→ XP creator was Kent Beck, one of the authors of agile manifesto

→ Kent Beck developed XP over the course of his consulting to Smalltalk projects in 80s and 90s

What is XP

→ Software development methodology

→ Frequent releases in short development cycles

→ Good engineering practices to produce quality software

When to use XP

→ When you have dynamically changing software requirements

→ When you are using new technology

→ When the technology you are using allows for automated unit and functional tests

→ When you have small co-located teams in the face of vague and changing requirements

XP Values

→ Simplicity

→ Only do what is needed and asked for the user to avoid waste

→ Communication

→ Everyone is part of the team, communication daily and face to face is better

→ Feedback

→ Demonstrate software early and often and listen to feedback

→ Courage

→ Tell the truth about progress and estimates

→ Respect

→ Respect each other to order to communicate with each other

XP Practices

→ Shared understanding

→ The planning game (User stories, sizing, releases)

→ Metaphor(goals)

→ Collective ownership

→ Coding standards

→ Simple design

→ Continuous process

→ Small releases

→ Refactoring

→ Rapid feedback

→ Testing

→ Pair programming

→ Continuous integration

→ On-site customer

→ Developer welfare

→ 40 hour work week

XP Summary

→ Lightweight software development technology

→ For vague and changing requirements

→ Medium sized teams

→ Discipline, humanistic

→ Adaptable

→ Roles

→ Values

→ Practices

→ to encourage interaction

XP Benefits

→ Improves productivity

→ Reduce project risk

→ Improve responsiveness to business changes

→ Produces high quality software

→ Adds fun to building software in teams

→ Accurate and frequent feedback about progress

→ Many opportunities to dramatically change requirements

→ A smaller initial investment

→ The opportunity to go faster

XP Weaknesses

→ Difficulty coordinating larger teams

→ Can result in a never ending project if not managed properly

→ Tendency to not document throughout the project

→ Predicting the precise features to accomplish in a fixed time/budget
